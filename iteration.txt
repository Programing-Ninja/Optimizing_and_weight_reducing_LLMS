--- EXPERIMENT FILE ---

File: /scratch/DA24B039/sct_tq/experiments/sct_tq_pareto.py

What it does:
  Sweeps SCT energy {None=dense, 0.99, 0.95, 0.90, 0.80} x TurboQuant KV configs
  {fp16-full, (k=4,v=4), (k=3,v=4), (k=3,v=2), (k=2,v=2)} on SmolLM2-135M.

  SCT: replaces MLP nn.Linear layers (gate_proj/up_proj/down_proj) with
  SpectralLinear.from_linear() at the given energy threshold. This does a
  truncated SVD and stores W = U diag(s) V^T. At 135M scale this rarely
  compresses — see COUNTERINTUITIVE section below.

  KV compression: uses a DynamicLayer subclass (TurboQuantLayer) that intercepts
  the transformers 5.x cache layer update() call, compresses keys via
  TurboQuantProd and values via quantize_values (group quantization), stores
  compressed tensors, and returns dequantized full-history K and V for attention.
  The Cache subclass approach WORKED — no fallback needed.

  Per config: measures perplexity, weight bytes, compressed KV bytes, peak RSS,
  and tokens/sec. Computes Pareto frontier (non-dominated on all 4 axes).
  Saves results to experiments/sct_tq_pareto_results.json.
  Saves scatter plot to experiments/sct_tq_pareto.png.

Key modeling decisions (rule 6 - flag the counterintuitive):
  - The whole insight hinges on an ASYMMETRY: SCT error is a structured BIAS that
    falls ~LINEARLY in retained energy (Eckart-Young: ||W-W_r||^2=(1-eta)||W||^2),
    while TurboQuant is UNBIASED so its error is pure VARIANCE that falls
    ~EXPONENTIALLY in bits (2^(-2b)). Linear-vs-exponential is exactly what makes
    the JOINT optimum nontrivial: at tight budgets spend bits on TQ first, at loose
    budgets push TQ to 4-bit early and invest the rest in SCT energy; the crossover
    budget is the optimum we want.
  - Additivity dL ~ alpha(1-eta)+beta*2^(-2b) is an ASSUMPTION, not a fact. SCT acts
    on static weights, TQ on the dynamic cache, so the cross term should be 2nd-order
    -- but if compressing weights shifts the cache statistics it could couple. The
    toy combined-grid arm is specifically there to confirm or REFUTE this; a large
    cross term would itself be the headline result.
  - recovery-LoRA enters the model as an alpha-SHRINK (it folds into attention
    projections and recovers SCT's MLP bias -- consistent with the earlier note that
    peft can't wrap SpectralLinear). Open Q: does that push the optimal eta upward?

Why a toy linear-attention + linear-MLP on Gaussian: it makes the loss an exact
quadratic with a KNOWN curvature H, so the predicted "excess loss ~ 1/2 tr(dW^T H dW)"
becomes checkable as an equality, not a hand-wave -- cheap, on a laptop, before
spending A100-hours.

WHAT DIDN'T MATCH THE DOC (findings to fold back into THEORETICAL_ANALYSIS.md)
  1. ΔL is CONCAVE in (1-η), not linear: small-σ (last-discarded) modes carry
     disproportionate loss-curvature. So α is a LOCAL slope; global through-
     origin R²=0.37. The α(1-η) model is only valid small-perturbation.
  2. TurboQuant effective exponent p≈1.83, NOT the asymptotic 2. The codebook's
     own finite-rate Lloyd-Max distortion has the same slope. Part A now uses
     ΔL_TQ = β_p·2^(−p·b) with measured p.
  3. Additivity is regime-dependent: SUB-additive coupling up to 46% at the
     aggressive corner (η=0.9,b=2). Sign is negative — compressing weights first
     lowers KV rank so quantization noise costs LESS loss. Beneficial for the
     joint budget (Part E q1 answered: coupled at aggressive compression).
  4. TERMINOLOGY (workspace rule 6): §A.3 "shift η upward (compress harder)" is
     inverted — η is retained energy, so harder compression = η DOWN. Solver
     confirms the intent (η* 0.708->0.640); doc wording should be fixed.
  5. Anisotropic Σ_x: α_aniso=3.03e-3 with poor local R²=0.30 — curvature-
     weighting is much stronger, linear α model degrades further.

CAVEAT ON PART A BYTE MODEL

  makes the optimum degenerate (η->1). Solver demo therefore uses a BALANCED

  weight:KV byte ratio is a deployment knob (model size × context length) — the
  solver itself is scale-agnostic; feed real (α,β,byte-curves) when HPC returns.

ROOT-CAUSE FOUND (profiled, not guessed): quantized-KV points were CPU-bound,
not GPU-bound or eval-subset-bound.
  - Live profiling on GPU 1 (Llama-3.1-8B, already cached, local_files_only,
    no HF_TOKEN needed): during a quantized-KV forward, GPU showed 0%
    utilization while the process pinned ~2640% CPU (26-32 of 32 cores).
  - Microbenchmark isolated the cause: torch.linalg.qr on a 128x128 matrix
    (the rotation-matrix construction inside TurboQuantProd.__init__) costs
    155.7ms/call at torch's default 32-thread intra-op pool on this node,
    vs 0.43ms/call single-threaded — a 340x THREAD-POOL-OVERHEAD penalty on
    a matrix so tiny the actual FLOPs are negligible.
  - make_cache_factory (pipeline/eval_tasks.py) built a fresh TurboQuantCache
    per forward call (cache = cache_factory() inside eval_perplexity AND
    inside _continuation_logprob, called once per hellaswag/mmlu/truthfulqa
    choice). Each fresh cache meant fresh TurboQuantLayer instances for all
    32 layers, each lazily rebuilding its TurboQuantProd (rotation + QJL
    matrices) from scratch. ~32 layers x ~190 forwards/point (quick subset
    sizes) = ~6080 redundant QR decompositions x 155.7ms ~ 947s — accounts
    for essentially all of the measured 998s/1072s per point.

FIX (pipeline/tq_cache.py, pipeline/eval_tasks.py):
  - TurboQuantLayer takes an optional quantizer_cache dict, keyed by
    (seed_offset, device); _get_quantizer() checks/populates it instead of
    always rebuilding. Lazy (built on first real forward) so per-layer
    device inference under multi-GPU dispatch (the 70B --big-model path)
    still works correctly.
  - TurboQuantCache passes quantizer_cache through to its layers.
  - make_cache_factory builds ONE shared quantizer_cache dict per eval point
    (per distinct key_bits/value_bits config) and closes over it, so every
    forward within that point's evaluate_all() call reuses the same 32
    per-layer quantizers instead of rebuilding them.
  - No numerical change: same seeds, same construction params. VERIFIED via
    a standalone profiling re-run: dense/KV=3x2 ppl=13.19 both before and
    after the fix (identical to 4 sig figs), confirming behavior-preserving.
