=== PLAN: SCT x TurboQuant joint Pareto frontier (CLAUDE.md rule 3) ===
Date: 2026-06-15   Author model: Opus (planning) -> Sonnet (implementation)

GOAL
----
Combine SCT (training/weight-side compression) with TurboQuant (inference KV-cache
compression) on ONE model, sweep the knobs, and trace the Pareto frontier across
four axes: (1) compression, (2) RAM usage, (3) latency, (4) accuracy.

ENVIRONMENT DECISION
--------------------
- Existing SCT/env (conda py3.10): torch import FAILS (CXXABI_1.3.15 not found ->
  loads system libstdc++ instead of conda's). NOT usable as-is.
- Base conda py3.13: no torch.
- GPU: nvidia-smi "Driver/library version mismatch" -> CUDA unusable. CPU-ONLY.
- ACTION: build a fresh venv at experiments/.venv (or sct_tq/.venv) from conda
  python, install CPU torch + transformers + datasets + numpy + matplotlib, then
  `pip install -e` both SCT and turboquant. Ensure conda libstdc++ is found
  (LD_LIBRARY_PATH=/opt/miniconda3/lib) -> wrap run in a small launcher if needed.

EXPERIMENT DESIGN  (experiments/sct_tq_pareto.py)
-------------------------------------------------
Model: HuggingFaceTB/SmolLM2-135M (CPU-feasible, matches existing examples). Configurable.

Knobs swept:
  - SCT energy E in {none(dense), 0.99, 0.95, 0.90, 0.80}
      * energy controls adaptive rank of SpectralLinear on MLP layers (from_linear).
      * "none" = unmodified dense model (baseline point).
  - TQ KV config in {none(fp16), (key_bits=4,val_bits=4), (3,4), (3,2), (2,2)}
      * Implement a transformers Cache subclass `TurboQuantCache` that compresses
        K via TurboQuantProd, V via group quantize_values on update(), stores
        compressed, returns dequantized K/V for attention. Gives REAL perplexity
        impact + REAL compressed byte counts.

Per (energy x TQ) config, measure:
  - compression: weight_bytes (SCT factors vs dense) + peak KV_bytes (TQ compressed
    vs fp16 baseline) -> overall bytes & ratio vs dense-fp16.
  - RAM: peak process RSS during a fixed eval workload (resource.getrusage / psutil).
  - latency: tokens/sec (prefill+decode) over a fixed generation length.
  - accuracy: perplexity on a fixed held-out text slice (wikitext-2 sample or the
    alpaca slice already used by examples). Lower = better.

Output:
  - experiments/sct_tq_pareto_results.json  (all points + config)
  - Pareto frontier computed for (accuracy vs each cost axis); printed table.
  - Optional matplotlib scatter w/ frontier -> experiments/sct_tq_pareto.png.

Default fast mode: NO finetune (measure post-hoc truncation+quant quality). Flag
--finetune-steps N to optionally recover SCT accuracy per energy.

COUNTERINTUITIVE NOTES (CLAUDE.md rule 6) -- to surface to user:
  1. SmolLM2-135M hidden dim = 576 (< 2048). Per CLAUDE.md, SCT compresses best at
     hidden >= 2048; at 135M, high-energy SCT yields little/negative weight savings.
     Pareto will show this. Model is configurable to a 1.7B model if a real win is
     wanted (but heavy on CPU).
  2. TQ hybrid decode dequantizes history to float32 -> KV MEMORY savings are real,
     but COMPUTE/latency savings are NOT realized (may add overhead). Latency axis
     mostly reflects SCT MLP FLOP reduction + TQ overhead. Report honestly.

=== PLAN v2: Systematic SCT HP-tuning + 1.7B (CLAUDE.md rule 3) ===
Date: 2026-06-15   Opus(plan) -> Sonnet(impl/run)

WHY: full sweep showed SCT post-hoc (no finetune) is useless (ppl explodes) AND
gives no weight compression at 135M (hidden=576 < 2048). To put SCT on the
frontier we must (a) finetune with the RIGHT recipe, (b) move to 1.7B (hidden=2048).

KNOWN LEVER (from SCT/examples/sct_per_component_lr.ipynb): PER-COMPONENT LR.
  - Uniform LR -> SCT ppl ~65-87 (bad).
  - Two param groups: {q,k,v,o_proj, *norm} @ LOW_LR (~2e-5) ; {SCT U,s,V} @
    HIGH_LR (~5e-4), ratio ~25x -> closes gap (target ppl <10).
  - Unfreeze attention projections too (not just norms). Retract after every step.

PHASE 1 -- HP tuning on SmolLM2-135M (cheap, do exhaustively):
  Implement real per-component-LR finetune in the experiment code (param groups:
  sct_factors / attn+norm-dense / rest-frozen). Grid (small, principled):
    sct_lr   in {1e-3, 5e-4, 1e-4}
    lr_ratio in {25x, 10x}  (dense_lr = sct_lr / ratio)
    unfreeze in {norms_only, norms+attn}
    energy   in {0.90, 0.95}
    steps    = 300 (fixed), batch 4, seq 128, alpaca (as examples)
  Metric: post-finetune perplexity on held-out eval. Pick BEST recipe.
  Output: experiments/sct_hp_tune_135m.json + best recipe printed.
  NOTE: even best recipe yields ~negative weight compression at 135M -> Phase 1
  is for the RECIPE, not the Pareto.

PHASE 2 -- SmolLM2-1.7B with best recipe + joint TQ Pareto:
  - FIRST: probe 1.7B -- load (fp32) RAM, time ONE finetune step on CPU, report
    ETA for {200,300,500} steps. Do NOT blindly run hours.
  - Then finetune SCT-1.7B at best recipe (capped steps per ETA/budget), apply TQ
    KV configs, measure the 4-axis Pareto (compression/RAM/latency/accuracy).
  - Output: experiments/sct_tq_pareto_1p7b.json (+ png).

CONSTRAINTS / COUNTERINTUITIVE (rule 6):
  - GPU dead -> CPU only. 1.7B finetune is SLOW; step budget is the lever, gate on
    measured ETA before running the long job.
  - 135B... no: 135M can never show good weight compression; don't judge SCT there.
  - Per-component LR is the whole ballgame for SCT accuracy -- center tuning there.

STEPS: 1[done] plan+prompts. 2 implement per-component finetune + HP harness.
  3 run Phase1 135M grid (background). 4 probe 1.7B ETA. 5 report + (gated) run
  Phase2. 6 iteration.txt + README + commit/push.
