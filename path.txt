GOAL
----
Combine SCT (training/weight-side compression) with TurboQuant (inference KV-cache
compression) on ONE model, sweep the knobs, and trace the Pareto frontier across
four axes: (1) compression, (2) RAM usage, (3) latency, (4) accuracy.

  (LD_LIBRARY_PATH=/opt/miniconda3/lib) -> wrap run in a small launcher if needed.

EXPERIMENT DESIGN  (experiments/sct_tq_pareto.py)
-------------------------------------------------
Model: HuggingFaceTB/SmolLM2-135M (CPU-feasible, matches existing examples). Configurable.

Knobs swept:
  - SCT energy E in {none(dense), 0.99, 0.95, 0.90, 0.80}
      * energy controls adaptive rank of SpectralLinear on MLP layers (from_linear).
      * "none" = unmodified dense model (baseline point).
  - TQ KV config in {none(fp16), (key_bits=4,val_bits=4), (3,4), (3,2), (2,2)}
      * Implement a transformers Cache subclass `TurboQuantCache` that compresses
        K via TurboQuantProd, V via group quantize_values on update(), stores
        compressed, returns dequantized K/V for attention. Gives REAL perplexity
        impact + REAL compressed byte counts.

Per (energy x TQ) config, measure:
  - compression: weight_bytes (SCT factors vs dense) + peak KV_bytes (TQ compressed
    vs fp16 baseline) -> overall bytes & ratio vs dense-fp16.
  - RAM: peak process RSS during a fixed eval workload (resource.getrusage / psutil).
  - latency: tokens/sec (prefill+decode) over a fixed generation length.
  - accuracy: perplexity on a fixed held-out text slice (wikitext-2 sample or the
    alpaca slice already used by examples). Lower = better.

Output:
  - experiments/sct_tq_pareto_results.json  (all points + config)
  - Pareto frontier computed for (accuracy vs each cost axis); printed table.
  - Optional matplotlib scatter w/ frontier -> experiments/sct_tq_pareto.png.

Default fast mode: NO finetune (measure post-hoc truncation+quant quality). Flag
--finetune-steps N to optionally recover SCT accuracy per energy.

COUNTERINTUITIVE NOTES (CLAUDE.md rule 6) -- to surface to user:
  1. SmolLM2-135M hidden dim = 576 (< 2048). Per CLAUDE.md, SCT compresses best at
     hidden >= 2048; at 135M, high-energy SCT yields little/negative weight savings.
     Pareto will show this. Model is configurable to a 1.7B model if a real win is
     wanted (but heavy on CPU).
  2. TQ hybrid decode dequantizes history to float32 -> KV MEMORY savings are real,
     but COMPUTE/latency savings are NOT realized (may add overhead). Latency axis
     mostly reflects SCT MLP FLOP reduction + TQ overhead. Report honestly.

=== PLAN v2: Systematic SCT HP-tuning + 1.7B (CLAUDE.md rule 3) ===
Date: 2026-06-15   Opus(plan) -> Sonnet(impl/run)

WHY: full sweep showed SCT post-hoc (no finetune) is useless (ppl explodes) AND
gives no weight compression at 135M (hidden=576 < 2048). To put SCT on the
frontier we must (a) finetune with the RIGHT recipe, (b) move to 1.7B (hidden=2048).

KNOWN LEVER (from SCT/examples/sct_per_component_lr.ipynb): PER-COMPONENT LR.
  - Uniform LR -> SCT ppl ~65-87 (bad).
  - Two param groups: {q,k,v,o_proj, *norm} @ LOW_LR (~2e-5) ; {SCT U,s,V} @
    HIGH_LR (~5e-4), ratio ~25x -> closes gap (target ppl <10).
  - Unfreeze attention projections too (not just norms). Retract after every step.

PHASE 1 -- HP tuning on SmolLM2-135M (cheap, do exhaustively):
  Implement real per-component-LR finetune in the experiment code (param groups:
  sct_factors / attn+norm-dense / rest-frozen). Grid (small, principled):
    sct_lr   in {1e-3, 5e-4, 1e-4}
    lr_ratio in {25x, 10x}  (dense_lr = sct_lr / ratio)
    unfreeze in {norms_only, norms+attn}
    energy   in {0.90, 0.95}
    steps    = 300 (fixed), batch 4, seq 128, alpaca (as examples)
  Metric: post-finetune perplexity on held-out eval. Pick BEST recipe.
  Output: experiments/sct_hp_tune_135m.json + best recipe printed.
  NOTE: even best recipe yields ~negative weight compression at 135M -> Phase 1
  is for the RECIPE, not the Pareto.

PHASE 2 -- SmolLM2-1.7B with best recipe + joint TQ Pareto:
  - FIRST: probe 1.7B -- load (fp32) RAM, time ONE finetune step on CPU, report
    ETA for {200,300,500} steps. Do NOT blindly run hours.
  - Then finetune SCT-1.7B at best recipe (capped steps per ETA/budget), apply TQ
    KV configs, measure the 4-axis Pareto (compression/RAM/latency/accuracy).
  - Output: experiments/sct_tq_pareto_1p7b.json (+ png).

CONSTRAINTS / COUNTERINTUITIVE (rule 6):
  - GPU dead -> CPU only. 1.7B finetune is SLOW; step budget is the lever, gate on
    measured ETA before running the long job.
  - 135B... no: 135M can never show good weight compression; don't judge SCT there.
  - Per-component LR is the whole ballgame for SCT accuracy -- center tuning there.

  3 run Phase1 135M grid (background). 4 probe 1.7B ETA. 5 report + (gated) run

=== Date: 2026-06-26   Opus(plan)                                            ===

DECISIONS (confirmed with user 2026-06-26):
  - Hardware:  single NVIDIA A100 40GB  (bf16 8B ~16GB fits + LoRA + eval).
  - Model:     meta-llama/Llama-3.1-8B  (BASE, gated -> needs HF_TOKEN). hidden=4096
               => SCT actually compresses here (the whole point vs SmolLM2-135M).
  - Eval:      SUBSET per benchmark for the sweep (fast). Optionally re-run full
               on frontier points later.
  - Recovery:  recovery-LoRA TOGGLE -> measure utility BOTH without and with LoRA
               so the plot shows the recovery gain. LoRA-scope is a flag
               (per-energy default; per-grid-point optional).
  - Source:    SCT, TurboQuant, and recovery-LoRA code will be IMPORTED by the user
               into the repo. Pipeline binds to their real APIs (NOT the inlined
               copies in experiments/sct_tq_pareto.py).

GOAL
----
Trace the Pareto frontier of UTILITY (y) vs COMPRESSION (x) for Llama-3.1-8B as we
sweep two compression knobs jointly:
  (A) SCT energy E  -> adaptive rank of SpectralLinear MLP factors (weight-side).
  (B) TurboQuant KV bits (key_bits, value_bits) -> KV-cache-side quantization.
And answer the two-way question:
  - How does weight-compression energy change the tolerable extent of quantization?
  - How does quantization change the achievable limit of weight compression?
  (i.e. interaction between A and B, visualized as frontier shift + a heatmap.)

UTILITY METRIC  (pipeline/utility.py)
-------------------------------------
Four components, each computed on a fixed SUBSET, on the SAME modified model so the
TQ-quantized K/V code path is exercised (see EVAL note):
  - Perplexity   : wikitext-2-raw-v1 (lower better)            -> ppl
  - HellaSwag    : length-normalized loglik acc (~1000 items)  -> acc_norm
  - MMLU         : 4-way loglik acc (~500 items, stratified)   -> acc
  - TruthfulQA   : MC2 (full ~817, it's small)                 -> mc2
Aggregate utility U in [0,1], normalized against the DENSE fp16 baseline so dense~1:
    s_ppl  = baseline_ppl / ppl                 (relative, capped at 1.0)
    s_hs   = acc_norm / baseline_acc_norm
    s_mmlu = acc       / baseline_mmlu
    s_tqa  = mc2       / baseline_mc2
    U = w_ppl*s_ppl + w_hs*s_hs + w_mmlu*s_mmlu + w_tqa*s_tqa   (default equal w)
  Weights configurable. Store raw + normalized + U in JSON for transparency.

EVAL CODE PATH (critical, rule 6 note)
--------------------------------------
HellaSwag/MMLU/TruthfulQA/PPL are LOGLIKELIHOOD tasks = single forward, no decode.
Default HF/lm-eval forwards run use_cache=False, so a KV-cache quantizer would NEVER
be exercised and quantization would look free. To make TQ's quality impact REAL we
force every eval forward with use_cache=True + a TurboQuantCache, so attention sees
compressed->dequantized K/V even in prefill. => custom lightweight evaluator
(pipeline/eval_tasks.py) rather than stock lm-eval, for full control of the cache
path. Scoring follows standard recipes (length-normalized gold-continuation loglik).

COMPRESSION METRIC  (pipeline/compression.py)
---------------------------------------------
  weight_bytes = SCT factor params (U,s,V) on compressed MLPs + remaining dense
                 params + (merged recovery-LoRA params if applied) at storage dtype.
  kv_bytes     = peak TurboQuant compressed KV bytes for the eval workload
                 (vs fp16 baseline 2*L*Hkv*hd*T*2).
  total_bytes  = weight_bytes + kv_bytes.
  compression_ratio = dense_fp16_total_bytes / total_bytes  (x-axis).

SWEEP GRID  (run_pareto.py)
---------------------------
  energies = [dense, 0.99, 0.97, 0.95, 0.90, 0.85]
  tq_kv    = [fp16, (4,4), (3,4), (3,2), (2,2)]
  lora     = [off, on]            # toggle
  => for each (energy, tq, lora) record {raw metrics, U, bytes, ratio, ram, tok/s}.
  Frontier computed over (U, compression_ratio); also a 2-D heatmap U(energy, kv).
  --quick mode: 2 energies x 2 kv x lora{off,on} for smoke test.

RECOVERY-LoRA  (pipeline/recovery.py wraps imported recovery-LoRA)
-----------------------------------------------------------------
  After SCT compression (per energy), train a recovery-LoRA on a small instruct/text
  mix to recover utility; default scope = per-energy (reused across kv configs),
  --lora-scope per-point optional. Toggle records U_before vs U_after.

MODULES (new pipeline/ package; reuse imported SCT/TQ/LoRA, do NOT re-inline)
  pipeline/model_build.py  - load Llama bf16, apply SCT (imported), apply LoRA,
                             build TurboQuantCache (imported).
  pipeline/eval_tasks.py   - ppl + HellaSwag + MMLU + TruthfulQA loglik w/ TQ cache.
  pipeline/utility.py      - normalization + aggregate U.
  pipeline/compression.py  - byte accounting + ratio.
  pipeline/recovery.py     - recovery-LoRA train/load wrapper.
  pipeline/pareto.py       - frontier + scatter plot + energy-vs-kv heatmap.
  run_pareto.py            - orchestrator (grid, JSON, PNGs).
  run.sh / README          - A100 run instructions (HF login, deps, invocation).

A100 40GB MEMORY PLAN
  - Load base in bf16 (~16GB). SCT factorization done layer-by-layer on GPU/CPU to
    avoid a 2nd dense copy. LoRA adapters tiny. Eval batch sized to fit (guarded).
  - One model in memory at a time; free + empty_cache between energies.
  - device_map kept simple/single-GPU but configurable.
